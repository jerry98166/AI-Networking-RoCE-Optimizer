## 2. AI 訓練網路背景：為什麼這題重要

### 2.1 分散式訓練的流量特徵

AI 訓練常見操作像 all-reduce、broadcast、parameter sync，會形成：
1. 高吞吐長流。
2. 同步階段突發流量。
3. 對延遲抖動很敏感。

關鍵不只在頻寬，而是穩定性。
如果某些節點網路延遲突然變高，整個 step 會卡在同步等待，導致 GPU 閒置。

### 2.2 為什麼傳統丟包重傳不理想

TCP 型邏輯偏向「丟包後再補救」，在 AI 訓練這種高頻同步場景，代價是：
1. 重傳等待使尾延遲惡化。
2. 吞吐波動放大，影響 step time 穩定。
3. 在高負載下更容易惡性循環。

所以你的方向是合理的：
盡量在「丟包前」就先感知擁塞，讓傳送端有機會提前調整。

## 3. RoCE v2 核心知識

### 3.1 RDMA 與 RoCE 的關係

RDMA 是一種「讓主機能低負擔進行記憶體資料搬移」的機制。
RoCE 是把 RDMA 思想放到乙太網環境中實作的一種方式。

RoCE v2 的重點：
1. 走在 IP + UDP 之上（可跨 L3）。
2. 兼顧資料中心實務部署彈性。
3. 常搭配擁塞控制機制使用。

### 3.2 你面試最常被問的 RoCE 問題

問：為何 RoCE v2 對 AI 網路有吸引力？
答：低延遲、低 CPU 開銷、高吞吐，且可融入資料中心 IP 網路。

問：為何還需要 ECN？
答：即使用 RoCE，在高負載時仍會有擁塞風險。ECN 可把「擁塞資訊」顯性化，提升控制迴路反應速度。

## 4. ECN 完整教學

### 4.1 ECN 是什麼

ECN 是 Explicit Congestion Notification，核心想法是：
不必等到丟包才知道塞車，而是在路徑出現擁塞徵兆時先標記封包。

### 4.2 直觀理解

把網路想成高速公路：
1. 傳統：塞爆才封路（丟包）。
2. ECN：快塞時先亮燈警示（標記）。

亮燈後，端點可以早一點降速，避免真正撞車（溢位丟包）。

### 4.3 在你專題中的 ECN 落地

對應 [src/roce_ecn_marker.v](src/roce_ecn_marker.v)：
1. 模組接收 AXI-Stream 封包。
2. 根據擁塞觸發條件決定是否標記。
3. 維持封包通行與握手正確性。

這表示你不是只講概念，而是已把概念變成可執行 RTL。

## 5. 封包與欄位定位基礎

### 5.1 為什麼欄位定位困難

你現在是 128-bit 資料寬度，封包是逐拍進入。
欄位位置會受這些因素影響：
1. 資料端序（byte ordering）。
2. 每拍對齊起點。
3. 可變長度 header（例如 IPv4 options）。
4. 多拍跨界。

### 5.2 你該如何面試回答這題

可用這個框架：
1. 先做固定封包格式驗證，確認基礎定位。
2. 再做跨拍定位與 alignment correction。
3. 最後加可變長 header 支援與健壯性測試。

這會讓面試官知道你理解「原型」與「產品級」的差距。

## 6. AXI-Stream 必修

### 6.1 四個關鍵訊號

1. tdata：資料本體。  
2. tvalid：上游宣告資料有效。  
3. tready：下游宣告可接收。  
4. tlast：封包最後一拍。

### 6.2 握手定律

只有在 tvalid 與 tready 同時為 1 的時鐘邊緣，資料才真正被傳遞。

### 6.3 你一定要會講的工程意義

1. 不檢查握手就轉資料，會有資料錯位風險。  
2. tready 反壓不正確，會造成溢位或隱性丟包。  
3. tlast 處理錯，封包邊界會壞掉，後面全錯。

### 6.4 與你程式碼對應

在 [src/roce_ecn_marker.v](src/roce_ecn_marker.v) 與 [sim/tb_roce_enc_marker.v](sim/tb_roce_enc_marker.v)，你已經在做握手驅動與封包結束流程測試，這是非常正確的方向。

## 7. Verilog 設計基礎與常見坑

### 7.1 同步邏輯與組合邏輯

1. 同步邏輯：在時鐘邊緣更新狀態與寄存器。  
2. 組合邏輯：即時根據輸入計算下一狀態或控制訊號。

狀態機通常是：
1. next-state 組合邏輯。  
2. state register 同步邏輯。

### 7.2 Reset 原則

Reset 要讓所有關鍵寄存器回到可預期值：
1. state。  
2. valid/last。  
3. 內部計數器與暫存欄位。

否則波形會出現 X，難以 debug。

### 7.3 面試常問錯誤

1. 忘記處理 backpressure。  
2. 在未握手成功時改寫資料。  
3. 狀態機漏掉 default 或邊界條件。  
4. tlast 與資料對不齊。

你可直接回覆：
我會用握手檢查、assertion、以及 corner case testbench 來防止這些錯誤。

## 8. 狀態機教學：從你的模組學

### 8.1 為什麼要有 ST_IDLE / ST_HDR / ST_DATA

這是典型封包管線切分：
1. ST_IDLE：等待封包開始。  
2. ST_HDR：處理報頭欄位。  
3. ST_DATA：直通 payload，直到 tlast。

好處：
1. 職責清楚，容易定位 bug。  
2. 可在 ST_HDR 做協議判斷與標記。  
3. 後續擴充 monitor 比較容易。

### 8.2 你可以進一步優化的地方

1. 增加 error state，處理非法包。  
2. 增加 header parse done 訊號，避免跨拍競態。  
3. 增加 pipeline register，改善時序。

## 9. FIFO、優先級、排程

### 9.1 為什麼要優先級

AI 流量不是每種都同等重要。
如果所有流量走單一 FIFO，關鍵流可能被非關鍵流阻塞。

### 9.2 典型策略

1. Strict Priority：高優先級先出，簡單但可能餓死低優先級。  
2. WRR：加權輪詢，較公平。  
3. Aging：低優先級等待太久可升級，避免飢餓。

### 9.3 你專題可採用的回答

目前先用簡單優先策略建立功能正確性，再導入公平策略平衡 latency 與 starvation。

---

## 10. Flow Control 與 Backpressure 深入

### 10.1 背壓本質

背壓是把下游壓力透過 tready 反向傳遞。

### 10.2 為何這是零丟包核心

當下游忙時，如果上游還硬塞，FIFO 會爆。
因此只要你要談 0% loss，就一定要先談流控正確性。

### 10.3 面試可用一句話

沒有正確背壓，任何 ECN 或優先排程都只是短暫效果，系統最終仍會失穩。

## 11. 驗證方法學：從能跑到可信

### 11.1 三層驗證

1. 模組驗證：單一 RTL 功能正確。  
2. 子系統驗證：模組串接後介面一致。  
3. 系統驗證：負載與壓力下仍達 KPI。

### 11.2 testbench 要含哪些情境

1. Normal flow。  
2. Congestion trigger。  
3. Backpressure burst。  
4. 隨機封包長度。  
5. 邊界條件：最短包、最長包、連續 tlast。

### 11.3 自動化檢查建議

1. Assertion：握手與狀態機不變式。  
2. Scoreboard：比較輸入輸出封包一致性。  
3. Coverage：確認情境覆蓋率。

你若只看波形，容易漏 bug；自動檢查才是工程化關鍵。

## 12. KPI 與數學定義

### 12.1 延遲

若以 cycle 計算：
$Latency_{avg} = \frac{1}{N}\sum_{i=1}^{N}(t^{out}_i - t^{in}_i)$

### 12.2 吞吐

$Throughput = \frac{Total\ bits\ delivered}{Observation\ time}$

### 12.3 丟包率

$Loss\ Rate = \frac{Dropped\ packets}{Input\ packets}$

### 12.4 抖動

可用延遲標準差近似：
$Jitter \approx \sigma(Latency)$

面試價值：
你不只講定性改善，也能定量比較。
